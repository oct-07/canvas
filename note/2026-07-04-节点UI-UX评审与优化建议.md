# 节点 UI/UX 评审与优化建议

> 评审对象：`src/components/Canvas/**` 节点相关组件（`ImageNode`、`VideoNode`、`CustomPoint/**`、`FloatingEditor/**`、`index.jsx`）
> 评审时间：2026-07-04
> 评审版本：基于画布浮窗/选中交互重构后版本
> 评审角度：交互功能性 + UI 一致性 + 可维护性

---

## 0. 一句话总评

当前节点设计**侧重"画布能力"（磁吸/拖拽/3D 倾斜）但弱化"内容交互"（浮窗打开、数据回填、连线数据流）**。最影响体验的几个问题集中在 **`FloatingEditor` 没有走真正的"浮窗"定位**，靠的是固定屏幕坐标，导致节点和浮窗的视觉链接断裂。

```

    ├── 画布空白 → 添加图片/视频节点
    ├── 节点     → 节点操作（菜单项）
    └── 边       → 边操作（菜单项）
```

## 2. 评审问题清单

### 2.1 🔴 P0【交互/FloatingEditor】浮窗不是"真浮窗"，是固定居中弹窗

**证据** — `FloatingEditor/index.jsx` 第 56-91 行：

```js
position: "fixed",
left: "50%",
top: "calc(50% + 160px)",   // ← 固定死在中下方
transform: "translateX(-50%)",
width: 900,
height: 400,
```

**问题**：

1. 节点 A 在屏幕左上角 → 提示词框在中下方 → **视线在屏幕两端来回跳**
2. 节点 A 与 B 紧挨 → 提示词框**遮住 B 节点**
3. 不同屏幕尺寸、不同边栏开合 → 浮窗位置都不变 → 与节点**视觉链路断裂**
4. 多节点工作流 → 浮窗全覆盖画布，看不到其他节点

**额外浪费证据**：

- `nodeEditors[id].position` 字段在 `handleNodeClick` 里**计算了屏幕坐标**
- `handleNodesChange` 的 position 分支每次拖动**都调用 `setNodeEditorPosition`** 同步坐标

但 `FloatingEditor` 根本没用 `position` prop —— 直接 `top: calc(50% + 160px)` 写死。这个计算全部是**死代码**。

**建议**：

- 浮窗接受 `position` prop（屏幕绝对坐标）
- 节点侧 div 的 `position: absolute; left: 0; top: 100%` **改为** `position: fixed; left: ${nodeScreenX}px; top: ${nodeScreenY + nodeHeight + 8}px`
- 在 `utils/canvasEditor.js` 增加 `getAnchorPoint(node, viewport)` 工具函数，输出节点屏幕坐标
- 视口边界检测：靠近视口右/下边缘时，浮窗自动**翻到节点上方/左侧**

### 2.2 🔴 P0【数据回填】`FloatingEditor` 提交后强制清空输入

**证据** — `FloatingEditor/index.jsx` 第 118-120 行：

```js
onSubmit(submitData);
// 提交后清空输入
setPrompt("");
setImageUrl("");
```

**问题**：

- 用户提交成功后想"再微调提示词生成一次" → 提示词已被清空 → **必须重新写**
- 节点 `data` 已经被 `updateNodeData` 写入，但浮窗内的输入框是**非受控本地 state**（`useState`），不会回写

**建议**：

提交后通过 store 反向回填：

```js
onSubmit(submitData);
// 反向回填：让本地 state 跟随节点 data 重新渲染
useEffect(() => {
  if (!editor?.data) return;
  setPrompt(editor.data.prompt ?? "");
  setImageUrl(editor.data.imageUrl ?? "");
}, [editor?.data?.prompt, editor?.data?.imageUrl]);
```

或把 `prompt`/`imageUrl` 改成**完全受控**于 `editor.data`（推荐）。

### 2.3 🔴 P0【键盘】删除快捷键误触风险

**证据** — `index.jsx` 第 289-295 行：

```js
if (e.key === "Delete" || e.key === "Backspace") {
  if (e.target.tagName === "INPUT" || e.target.tagName === "TEXTAREA") {
    return;
  }
  const { selectedNodeId } = useCanvasStore.getState();
  if (selectedNodeId) removeNode(selectedNodeId);
}
```

**问题**：

- `contenteditable` 元素（`PromptInputArea` 可能用）**不在 INPUT/TEXTAREA 内**
- Ant Design 的 Select/TreeSelect/Modal 等组件的输入元素 **也不是 INPUT/TEXTAREA tag** —— 是 `<div contenteditable>` 或 `<span>`
- 用户在提示词框里按 Backspace 想删字符 → **直接删除节点**

**建议**：

```js
// utils/dom.js
export const isEditableElement = (el) => {
  if (!el) return false;
  const tag = el.tagName;
  if (tag === "INPUT" || tag === "TEXTAREA" || tag === "SELECT") return true;
  if (el.isContentEditable) return true;
  // Ant Design 常用交互元素
  if (el.closest(".ant-select, .ant-modal, .ant-popover, .ant-dropdown")) return true;
  return false;
};
```

### 2.4 🔴 P0【状态隔离】`clearSelection` 与浮窗状态耦合不清

**证据** — `index.jsx` 第 159-163 行：

```js
const handlePaneClick = useCallback(() => {
  clearSelection();
  hideContextMenu();
  hideActiveEditor();  // ← 调用时不传参，等于清空所有
}, [clearSelection, hideContextMenu, hideActiveEditor]);
```

**问题**：

- `clearSelection` 是 React Flow 的清空 selection，会触发 `nodes` 数组的 `selected: false` 更新
- 当前 `handleNodesChange` 不再处理 select change，所以没问题
- **但**：`clearSelection` 与 `setNodes` 配合使用时，可能触发死循环（已在重构后避免，需继续保持）

**附加**：`selectedNodeId`（右键 setSelectedNode 后）与 React Flow `selected` 是两条独立 channel，详见 2.10。

### 2.5 🟡 P1【状态】`isFullScreen` 是 `FloatingEditor` 局部 state

**证据** — `FloatingEditor/index.jsx` 第 18-26 行：

```js
const [isFullScreen, setIsFullScreen] = useState(false);
```

**问题**：

- 用户在节点 A 全屏提示词框 → 切到节点 B → 全屏标记丢失，体验割裂
- 刷新页面 → 全屏标记丢失（这个无所谓）

**建议**：

把 `isFullScreen` 写到 `nodeEditors[id]` 里：

```js
nodeEditors: {
  [id]: { visible, nodeType, position, data, isFullScreen }
}
```

### 2.6 🟡 P1【键盘】缺少 `Esc` 关闭浮窗

**证据** — `index.jsx` keydown 监听没有 `Escape` 分支。

**问题**：用户对 Modal/Dropdown 的肌肉记忆是 `Esc` 关闭。当前面板只能点 × 或点空白处。

**建议**：在 keydown 监听里增加：

```js
if (e.key === "Escape") {
  if (useCanvasStore.getState().activeNodeId) {
    useCanvasStore.getState().hideActiveEditor();
    e.preventDefault();
  }
}
```

### 2.7 🟡 P1【代码质量】`draggedNodeIdRef` 死代码

**证据** — `index.jsx` 第 33、165-167、172-174 行：

```js
const draggedNodeIdRef = useRef(null);

const handleNodeDragStart = useCallback((_event, node) => {
  draggedNodeIdRef.current = node.id;
}, []);

const handleNodeDragStop = useCallback((_event, node) => {
  saveHistory();
  if (draggedNodeIdRef.current === node.id) {
    draggedNodeIdRef.current = null;   // ← 写完即清，无消费方
  }
}, [saveHistory]);
```

**问题**：

- ref 写入后由自己清空，**没有任何外部消费者**
- 整段逻辑可删除

**建议**：直接删除 ref + `handleNodeDragStart`/`handleNodeDragStop` 中的无关赋值。

### 2.8 🟡 P1【一致性】右键菜单触发的"选中"与 React Flow 选中错位

**证据** — `index.jsx` 第 141-148 行：

```js
const handleNodeContextMenu = useCallback(
  (event, node) => {
    event.preventDefault();
    useCanvasStore.getState().setSelectedNode(node.id);  // ← 业务态
    showContextMenu(event.clientX, event.clientY, "node", node.id);
  },
  [showContextMenu],
);
```

**问题**：

- `setSelectedNode` 写到 `selectedNodeId`（store 内部态）
- 但 React Flow 内部 `node.selected` 字段**没有同步**
- 结果：右键节点后，节点视觉上没有高亮，但 `selectedNodeId` 已被设置 → Delete 快捷键可工作但视觉不一致

**建议**：

- 要么把 `setSelectedNode` 改为同步 `nodes`（调用 `setNodes` 把 `selected` 字段同步）
- 要么右键前/后调用 `clearSelection` + 单独选中该节点

### 2.9 🟡 P1【Effect】`useEffect` 依赖项缺失（局部）

**证据** — `index.jsx` 第 265-320 行的 keydown `useEffect`：

```js
useEffect(() => {
  const handleKeyDown = (e) => {
    // ...
    if (e.key === "Delete") {
      const { selectedNodeId } = useCanvasStore.getState();  // getState 取最新值
      // ...
    }
    if (cmdKey && e.key === "c") {
      const { selectedNodeId, nodes: currentNodes, copyNode } = useCanvasStore.getState();
      // ...
    }
  };
  // ...
}, [
  undo, redo, canUndo, canRedo, clipboard, pasteNode,
  screenToFlowPosition, removeNode,
  // ← copyNode / selectedNodeId 未列
]);
```

**问题**：

- 当前实现通过 `getState()` 取最新值，**功能没问题**
- ESLint `react-hooks/exhaustive-deps` 会报警
- 维护时容易混淆"哪些值走 deps，哪些走 getState"

**建议**：要么全 `getState()`（移除 deps），要么全列 deps（effect 内用闭包值）。统一风格。

### 2.10 🟢 P2【视觉】选中态视觉反馈偏弱

**证据** — `ImageNode.jsx` 第 53-63 行：

```js
border: isTarget ? `2px solid ${magnetColor}`
  : isActive ? "2px solid #177ddc"
  : "1px solid #303030",   // ← 1→2px 差异很小
boxShadow: isTarget ? `0 0 0 2px ${magnetColor}66, 0 8px 24px ${magnetColor}44`
  : isActive ? "0 0 20px rgba(23, 125, 220, 0.3)"
  : "0 4px 12px rgba(0,0,0,0.3)",   // ← 蓝色阴影在深色背景上几乎不可见
```

**问题**：

- 边框粗细 1→2px 视觉差异微弱
- `rgba(23, 125, 220, 0.3)` 在 `#0d0d0d` 背景上几乎读不出
- 没有"被点亮"的瞬态反馈（点击瞬间到高亮态是 0.15s 渐变）

**建议**：

- 选中加 `outline: 2px solid #177ddc; outline-offset: 2px`（更显眼）
- 节点右上角显示 **"激活"小圆点**（区别于 React Flow selected 蓝框）
- 鼠标按下时给个 `transform: scale(0.98)` 缩放反馈（80ms）

### 2.11 🟢 P2【功能】节点功能入口不显式

**问题**：

- 节点上没有"提示词" / "参数"角标按钮
- 用户要"猜"点哪能开提示词框
- 没有 hover 显式按钮（hover 出现角标）

**建议**：

- 节点 hover / active 时，显示一个小图标按钮（如 `FormOutlined` / `EditOutlined`）
- 按钮点击 = `handleNodeClick` 的同样逻辑

### 2.12 🟢 P2【一致性】`ImageNode` 与 `VideoNode` 高度重复

**证据**：

| ImageNode.jsx | VideoNode.jsx |
|---|---|
| 边框判断 `isTarget/isActive/默认` | 完全一样 |
| box-shadow | 完全一样 |
| transition | 完全一样 |
| transform 3D 倾斜 | 完全一样 |
| `syncNodeDimensions` useEffect | 完全一样 |
| 内部内容：图片预览 / Icon `PictureOutlined` | 视频预览 / `PlayCircleOutlined` |

**问题**：

- 任何边框/阴影/激活逻辑调整**必须改两个文件** → 容易遗漏
- 90% 逻辑耦合 → 维护成本翻倍

**建议**（如允许重构）：

抽 `NodeShell` 容器组件：

```jsx
// CustomNode/NodeShell.jsx
const NodeShell = ({ id, selected, width, aspectRatio, children }) => {
  // ... 边框/阴影/3D 倾斜/激活 判断
  return <div style={shellStyle}>{children}</div>;
};
```

`ImageNode` / `VideoNode` 退化为**纯内容组件**，只关心渲染。

> ⚠️ 重构影响面较大，建议另起一个独立 sprint。

### 2.13 ⚪ 体验【一致性】嵌套圆角不一致

**证据**：

- `ImageNode.jsx` 第 52、86 行：`borderRadius: 12`（外）+ `borderRadius: 12px 12px 0 0`（内）
- `VideoNode.jsx` 第 57、94 行：`borderRadius: "12px"`（外）+ `borderRadius: "10px"`（内）

**建议**：统一外/内圆角（外 12 / 内 10），视觉更整齐。

### 2.14 ⚪ 体验【视觉】拖动光标无反馈

**证据** — `index.jsx` `handleNodeDragStart` 只设置 ref（**且是死代码**），没有视觉反馈。

**建议**：

```js
const handleNodeDragStart = useCallback(() => {
  document.body.style.cursor = "grabbing";
}, []);

const handleNodeDragStop = useCallback((_event, node) => {
  document.body.style.cursor = "";
  saveHistory();
}, [saveHistory]);
```

## 5. 一句话行动建议

**先做 P0 三条**（2.1 浮窗定位 / 2.2 受控回填 / 2.3 防误触），它们是当前用户体验上**最痛的三个点**：
1. 浮窗与节点的视觉链路断裂（每用一次都痛）
2. 提交后输入被清空（每提交一次都痛）
3. 在提示词框里按 Backspace 直接删了节点（每按一次都可能痛）

P0 完成后再做 P1（局部状态治理 / 一致性），最后视团队精力决定是否做 P2 的视觉加强与抽组件重构。
