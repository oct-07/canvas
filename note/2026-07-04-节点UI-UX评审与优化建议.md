## 0. 一句话总评

当前节点设计**侧重"画布能力"（磁吸/拖拽/3D 倾斜）但弱化"内容交互"（浮窗打开、数据回填、连线数据流）**。最影响体验的剩余痛点集中在 **`FloatingEditor` 没有走真正的"浮窗"定位**，靠的是固定屏幕坐标，导致节点和浮窗的视觉链接断裂。

**本次重构进度**（已落地的）：
- 节点拖动期间自动隐藏浮窗 + 拖动结束后自动打开（新增 `draggingNodeId` 全局态）
- `FloatingEditor` 常驻挂载（替换原条件渲染），刷新/重开不再丢失输入态
- **第二轮（13:30 落地）**：删除提交后残留的 2 行清空代码（修复闪烁）、新增 `utils/dom.js` 提供 `isEditableElement`（修复 Backspace 误触）、新增 Esc 关闭浮窗

## 2. 评审问题清单

### 2.1 🔴 P0【交互/FloatingEditor】浮窗不是"真浮窗"，是固定居中弹窗

**证据** — `FloatingEditor/index.jsx` 第 86-91 行：

```js
return {
  ...baseStyle,
  top: "calc(50% + 160px)",   // ← 固定死在中下方
  width: 900,
  height: 400,
};
```

**问题**：

1. 节点 A 在屏幕左上角 → 提示词框在中下方 → **视线在屏幕两端来回跳**
2. 节点 A 与 B 紧挨 → 提示词框**遮住 B 节点**
3. 不同屏幕尺寸、不同边栏开合 → 浮窗位置都不变 → 与节点**视觉链路断裂**
4. 多节点工作流 → 浮窗全覆盖画布，看不到其他节点

**额外浪费证据**：

- `nodeEditors[id].position` 字段在 `handleNodeClick` 里**计算了屏幕坐标**
- `handleNodesChange` 的 position 分支每次拖动**都调用 `setNodeEditorPosition`** 同步坐标

但 `FloatingEditor` 根本没用 `position` prop —— 直接 `top: calc(50% + 160px)` 写死。这个计算全部是**死代码**。

`src/utils/canvasEditor.js` 已有 `getNodeScreenPos(node, viewport)` 工具函数可复用。

**建议**：

- 浮窗接受 `position` prop（屏幕绝对坐标）
- 节点侧 div 的 `position: absolute; left: 0; top: 100%` **改为** `position: fixed; left: ${nodeScreenX}px; top: ${nodeScreenY + nodeHeight + 8}px`
- 视口边界检测：靠近视口右/下边缘时，浮窗自动**翻到节点上方/左侧**

### 2.2 ✅ 已实现【数据回填】`FloatingEditor` 提交后清空修复

**状态**：✅ **2026-07-04 13:30 落地** —— 删除了 `FloatingEditor/index.jsx` 中 `handleSend` 残留的 2 行清空代码（`setPrompt("")` / `setImageUrl("")`）。现有 `useEffect([editor?.data])` 回填链路已足够，无须在 `handleSend` 内手动清空。

**修改前** — `FloatingEditor/index.jsx` handleSend：

```js
onSubmit(submitData);
// 提交后清空输入
setPrompt("");
setImageUrl("");
```

**修改后**：

```js
onSubmit(submitData);
// 不再清空本地 state：editor.data 已通过 useEffect 回填到 setPrompt/setImageUrl
```

**保留的回填链路**（已存在） — `FloatingEditor/index.jsx` 第 46-53 行：

```js
useEffect(() => {
  if (!editor?.data) return;
  const data = editor.data;
  setPrompt(data.prompt ?? "");
  setStyleValue(data.style ?? "default");
  setImageUrl(data.imageUrl ?? "");
  setParams(data.params ?? params);
}, [editor?.data]);
```

**收益**：提交瞬间不再有"清空→回填"造成的微闪烁与多余 render，用户体验更顺滑。

### 2.3 ✅ 已实现【键盘】删除快捷键误触风险

**状态**：✅ **2026-07-04 13:30 落地** —— 新增 `src/utils/dom.js` 提供 `isEditableElement(el)`，keydown 已切换到该工具。

**新增文件** — `src/utils/dom.js`（核心实现）：

```js
export function isEditableElement(el) {
  if (!el || !el.tagName) return false;

  const tag = el.tagName;
  if (tag === "INPUT" || tag === "TEXTAREA" || tag === "SELECT") return true;
  if (el.isContentEditable) return true;

  // Ant Design 常用交互容器
  if (el.closest(".ant-select, .ant-tree-select, .ant-cascader, .ant-date-picker, .ant-time-picker, .ant-input-number, .ant-modal, .ant-popover, .ant-dropdown, .ant-tooltip, .ant-popconfirm")) {
    return true;
  }

  return false;
}
```

**keydown 改造** — `index.jsx`：

```js
if (e.key === "Delete" || e.key === "Backspace") {
  // 防止误触：输入框 / Ant 弹层 / contenteditable 内部应自行消费按键
  if (isEditableElement(e.target)) return;
  const { selectedNodeId } = useCanvasStore.getState();
  if (selectedNodeId) removeNode(selectedNodeId);
}
```

**覆盖范围**：INPUT / TEXTAREA / SELECT + contenteditable（PromptInputArea 已纳入）+ Ant Design 8 类交互容器。

### 2.4 🟢 P2【状态隔离】`clearSelection` 与浮窗状态耦合

**证据** — `index.jsx` 第 165-171 行：

```js
const handlePaneClick = useCallback(() => {
  clearSelection();
  hideContextMenu();
  hideActiveEditor();  // ← 调用时不传参，等于清空所有
}, [clearSelection, hideContextMenu, hideActiveEditor]);
```

**现状**：

- `clearSelection` 触发后，React Flow 的 `nodes` 数组 `selected: false` 变更
- 当前 `handleNodesChange` 已**不再处理 select 分支**（本次重构已修复），所以没有死循环风险
- 剩余问题仅为"`hideActiveEditor()` 不带参 = 清空所有节点的浮窗"，如果未来支持多浮窗同时显示，会被错误清空

**建议**：当前实现可保持，但建议加注释说明"`hideActiveEditor()` 无参调用是清屏语义"，避免后续维护误改。

### 2.5 🟡 P1【状态】`isFullScreen` 是 `FloatingEditor` 局部 state

**证据** — `FloatingEditor/index.jsx` 第 18-26 行：

```js
const [isFullScreen, setIsFullScreen] = useState(false);
```

**问题**：

- 用户在节点 A 全屏提示词框 → 切到节点 B → 全屏标记丢失，体验割裂
- 切换节点的瞬间，A 节点的 FloatingEditor 卸载（即使常驻挂载，但 React 会回收），下次切回 A 全屏标记不在

**建议**：把 `isFullScreen` 写到 `nodeEditors[id]` 里：

```js
nodeEditors: {
  [id]: { visible, nodeType, position, data, isFullScreen }
}
```

### 2.8 🟡 P1【一致性】右键菜单触发的"选中"与 React Flow 选中错位

**证据** — `index.jsx` 第 147-163 行：

```js
const handleNodeContextMenu = useCallback(
  (event, node) => {
    event.preventDefault();
    useCanvasStore.getState().setSelectedNode(node.id);  // ← 业务态
    showContextMenu(event.clientX, event.clientY, "node", node.id);
  },
  [showContextMenu],
);
```

**问题**：

- `setSelectedNode` 写到 `selectedNodeId`（store 内部态）
- 但 React Flow 内部 `node.selected` 字段**没有同步**
- 结果：右键节点后，节点视觉上没有高亮，但 `selectedNodeId` 已被设置 → Delete 快捷键可工作但视觉不一致

**建议**：

- 要么把 `setSelectedNode` 改为同步 `nodes`（调用 `setNodes` 把 `selected` 字段同步）
- 要么右键前/后调用 `clearSelection` + 单独选中该节点

### 2.9 🟡 P1【Effect】`useEffect` 依赖项缺失（局部）

**证据** — `index.jsx` 第 361-369 行的 keydown `useEffect` deps：

```js
}, [
  undo, redo, canUndo, canRedo, clipboard, pasteNode,
  screenToFlowPosition, removeNode,
  // ← copyNode / selectedNodeId / 当前 nodes 未列
]);
```

**问题**：

- 当前实现通过 `getState()` 取最新值，**功能没问题**
- ESLint `react-hooks/exhaustive-deps` 会报警
- 维护时容易混淆"哪些值走 deps，哪些走 getState"

**建议**：要么全 `getState()`（移除 deps），要么全列 deps（effect 内用闭包值）。统一风格。

### 2.10 🟢 P2【视觉】选中态视觉反馈偏弱

**证据** — `ImageNode.jsx` 第 55-66 行：

```js
border: isTarget ? `2px solid ${magnetColor}`
  : isActive ? "2px solid #177ddc"
  : "1px solid #303030",   // ← 1→2px 差异很小
boxShadow: isTarget ? `0 0 0 2px ${magnetColor}66, 0 8px 24px ${magnetColor}44`
  : isActive ? "0 0 20px rgba(23, 125, 220, 0.3)"
  : "0 4px 12px rgba(0,0,0,0.3)",   // ← 蓝色阴影在深色背景上几乎不可见
```

**问题**：

- 边框粗细 1→2px 视觉差异微弱
- `rgba(23, 125, 220, 0.3)` 在 `#0d0d0d` 背景上几乎读不出
- 没有"被点亮"的瞬态反馈（点击瞬间到高亮态是 0.15s 渐变）

**建议**：

- 选中加 `outline: 2px solid #177ddc; outline-offset: 2px`（更显眼）
- 节点右上角显示 **"激活"小圆点**（区别于 React Flow selected 蓝框）
- 鼠标按下时给个 `transform: scale(0.98)` 缩放反馈（80ms）

### 2.12 🟢 P2【一致性】`ImageNode` 与 `VideoNode` 高度重复

**证据**：

| ImageNode.jsx | VideoNode.jsx |
|---|---|
| 边框判断 `isTarget/isActive/默认` | 完全一样 |
| box-shadow | 完全一样 |
| transition | 完全一样 |
| transform 3D 倾斜 | 完全一样 |
| `syncNodeDimensions` useEffect | 完全一样 |
| 内部内容：图片预览 / Icon `PictureOutlined` | 视频预览 / `PlayCircleOutlined` |

**问题**：

- 任何边框/阴影/激活逻辑调整**必须改两个文件** → 容易遗漏
- 90% 逻辑耦合 → 维护成本翻倍

**建议**（如允许重构）：

抽 `NodeShell` 容器组件：

```jsx
// CustomNode/NodeShell.jsx
const NodeShell = ({ id, selected, width, aspectRatio, children }) => {
  // ... 边框/阴影/3D 倾斜/激活 判断
  return <div style={shellStyle}>{children}</div>;
};
```

`ImageNode` / `VideoNode` 退化为**纯内容组件**，只关心渲染。

> ⚠️ 重构影响面较大，建议另起一个独立 sprint。
